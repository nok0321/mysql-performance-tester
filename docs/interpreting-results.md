# 結果の読み方

## パーセンタイルの意味

| パーセンタイル | 意味 | 活用方法 |
|---|---|---|
| **P50**（中央値） | ユーザーの半数が経験するレイテンシ | 典型的なユーザー体験の指標 |
| **P75** | 上位25%が経験するレイテンシ | やや重いケースの把握 |
| **P90** | 上位10%が経験するレイテンシ | 遅延の傾向把握 |
| **P95** | 上位5%が経験するレイテンシ | **チューニングの主要ターゲット** |
| **P99** | 上位1%が経験するレイテンシ | アーキテクチャ上のボトルネック検出 |
| **P99.9** | 上位0.1%が経験するレイテンシ | 極端な外れ値（通常は参考値） |

> **P95 を最も重視してください。** P95 が悪い場合、20人に1人が遅いクエリを経験していることを意味します。

## レイテンシグレード（P95 基準）

| グレード | P95 レイテンシ | 評価 |
|---|---|---|
| **A+** | < 10ms | Excellent — 最適 |
| **A** | < 50ms | Very Good — 非常に良い |
| **B** | < 100ms | Good — 良い |
| **C** | < 200ms | Fair — 許容範囲 |
| **D** | < 500ms | Poor — 要改善 |
| **F** | ≥ 500ms | Critical — 至急改善が必要 |

---

## Buffer Pool ヒット率グレード

Buffer Pool はクエリ結果をメモリにキャッシュします。
ヒット率が高いほど、ディスク I/O を避けて高速に応答できます。

| グレード | ヒット率 | 評価 |
|---|---|---|
| **A+** | ≥ 99% | 最適 |
| **A** | ≥ 95% | 良好 |
| **B** | ≥ 90% | やや低い |
| **C** | ≥ 85% | 低い |
| **D** | ≥ 80% | 非常に低い |
| **F** | < 80% | 至急改善が必要 |

ヒット率が低い場合は `innodb_buffer_pool_size` の増加を検討してください。

---

## 変動係数（CV）の評価

変動係数（CV = 標準偏差 / 平均 × 100）は実行時間のばらつきを示します。

| CV | 評価 |
|---|---|
| < 10% | 非常に安定 |
| 10〜20% | 比較的安定 |
| 20〜30% | やや不安定 |
| > 30% | 不安定 — システム負荷・ネットワーク・ロック競合を確認 |

---

## ウォームアップの重要性

ウォームアップなしで計測した最初の数回は、Buffer Pool が cold な状態のため
実際より遅い値が出ます。ウォームアップを有効にすることで、
**本番環境の定常状態に近いレイテンシ**を計測できます。

```
ウォームアップあり:  P95 = 2.3ms  ← 本番に近い
ウォームアップなし:  P95 = 18.7ms ← 初回アクセスの遅延が混入
```

## ベストプラクティス

### 測定回数

```
推奨: 20 回以上
最低: 10 回（統計的信頼性の下限）
注意: 3〜5 回では外れ値の影響が大きく、P95/P99 が不安定になります
```

### 外れ値除去

外れ値除去は**デフォルトで有効（IQR法）**です。
GC・OS スケジューリングによるスパイクを除去し、より正確な分析ができます。

ただし「本番でたまに発生する遅延スパイクを含めて計測したい」場合は
外れ値除去を無効にしてください。

### 並列テストの戦略選択

| 戦略 | 特徴 | 適する用途 |
|---|---|---|
| **Random** | SQL をランダムに選んで実行 | 本番の混合ワークロードに近い条件 |
| **RoundRobin** | SQL を順番に均等実行 | 各クエリの公平な比較 |
| **Sequential** | 1種類の SQL を連続実行 | 単一クエリのスループット上限確認 |
| **CategoryBased** | カテゴリ別に分けて実行 | 読み書き分離環境のシミュレーション |

### テスト環境の準備

- 本番環境に近いデータ量でテストする
- テスト中は他の重い処理を停止してシステムリソースを確保する
- 複数回実行して結果の一貫性を確認する
- インデックス変更前後を比較する際は同条件・同データ量で比較する

## EXPLAIN ANALYZE の見方

EXPLAIN ANALYZE（MySQL 8.0.18+）は実際の実行計画と実測値を返します。

注目すべきポイント:

- **actual time**: 実際の実行時間（推定値とのズレが大きい場合は統計情報が古い）
- **rows examined**: スキャン行数（多すぎる場合はインデックスの追加を検討）
- **Using filesort / Using temporary**: 追加の並べ替え・一時テーブルが発生
- **Full table scan**: インデックスが使われていない（大テーブルでは致命的）
